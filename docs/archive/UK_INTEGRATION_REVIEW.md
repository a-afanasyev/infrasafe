## 1. Обзор проектов

### 1.1 InfraSafe
IoT-платформа мониторинга инфраструктуры многоквартирных домов. Собирает данные с контроллеров (промышленных ПК с датчиками), визуализирует на карте Leaflet, отслеживает электроснабжение, водоснабжение, отопление.

| Характеристика | Значение |
|---|---|
| Язык | JavaScript (Node.js 20+) |
| Фреймворк | Express.js |
| БД | PostgreSQL 15+ с PostGIS |
| Фронтенд | Vanilla JS, Leaflet.js, Chart.js |
| Развёртывание | Docker Compose, Nginx |
| Таблиц в БД | ~20 |
| Архитектура | Монолит (Controllers -> Services -> Models) |

### 1.2 UK Management Bot
Система управления заявками для управляющих компаний (УК) через Telegram-бот. Обрабатывает жизненный цикл заявок от жителей, управляет сменами исполнителей, оптимизирует назначения через ML-скоринг.

| Характеристика | Значение |
|---|---|
| Язык | Python 3.11+ |
| Фреймворк | Aiogram 3.x (Telegram), FastAPI (Web) |
| БД | PostgreSQL 15+ |
| ORM | SQLAlchemy 2.0 |
| Кэш | Redis |
| Развёртывание | Docker Compose (app + postgres + redis + web) |
| Таблиц в БД | 27 |
| Микросервисы | Media Service (отдельный стек) |

### 2.3 Комплементарность систем

**InfraSafe покрывает:** автоматический мониторинг (датчики, метрики, алерты) -- то, что происходит *до* создания заявки.

**UK Bot покрывает:** человеческий workflow (заявка, назначение, исполнение, приёмка) -- то, что происходит *после* обнаружения проблемы.

**Точка интеграции:** Алерт InfraSafe -> Заявка UK Bot.

## 3. Код-ревью UK Management Bot

### 3.1 Сильные стороны

1. **Продуманная middleware-цепочка:**
   ```
   DB Injection -> Auth -> Role Mode -> Shift Context
   ```
   Каждый слой отвечает за свою задачу, чистое разделение ответственности.

2. **Безопасность invite-токенов:**
   - HMAC-SHA256 подпись с `hmac.compare_digest()` (защита от timing-атак)
   - Отслеживание nonce для предотвращения повторного использования
   - Принудительная проверка INVITE_SECRET в production

3. **ML-оптимизация назначений:**
   - `AssignmentOptimizer` оценивает по 4 критериям (специализация, география, нагрузка, рейтинг)
   - `SmartDispatcher` автоматически назначает при score > 70
   - `WorkloadPredictor` прогнозирует нагрузку

4. **Structured logging:**
   - JSON-формат с фильтрацией sensitive данных (пароли, токены)
   - Компонентный контекст (`component="invite_service"`)

5. **Верификация пользователей:**
   - Многоуровневая проверка документов (паспорт, договор, квитанция)
   - Модерация администратором
   - Ограничение: максимум 2 верифицированных жителя на квартиру

### 3.2 Критические проблемы

#### P0: Безопасность CORS (FastAPI)
```python
# web/main.py
allow_origins=["*"],       # КРИТИЧЕСКАЯ УЯЗВИМОСТЬ
allow_credentials=True,
allow_methods=["*"],
```
**Проблема:** Разрешает запросы с любого домена с credentials. Позволяет CSRF-атаки на web-регистрацию.
**Решение:** Whitelist конкретных origins через env-переменную.

#### P0: Отсутствие HTML-экранирования
Текст заявок от пользователей отправляется в Telegram с `parse_mode=HTML` без экранирования спецсимволов `<`, `>`, `&`. Возможна инъекция HTML-тегов в сообщения бота.

### 3.3 Высокий приоритет

#### P1: Монолитные обработчики
| Файл | Строк | Проблема |
|---|---|---|
| `shift_management.py` | 3,677 | Смешивает UI, бизнес-логику, state management |
| `requests.py` | 3,262 | Дублирование логики отображения |
| `admin.py` | 2,691 | Пересечение с другими обработчиками |
| `user_management.py` | 2,494 | Длинные функции (100+ строк) |

**Рекомендация:** Извлечь в сервисы: `RequestDisplayService`, `StateTransitionService`, `AddressSelectionService`.

#### P1: Inconsistent error handling
- Широкие `except Exception as e` ловят всё
- Нет иерархии custom exceptions (`InviteExpiredError`, `ValidationError`, и т.д.)
- Функции возвращают то `False`, то error message, то raise exception

#### P1: Дублирование кода
- Логика выбора адреса/двора повторяется в 3+ обработчиках
- Определение языка пользователя дублируется
- Форматирование заявки для отображения повторяется в requests.py и admin.py

### 3.4 Средний приоритет

#### P2: Зависимости не зафиксированы
```
aiogram>=3.0.0    # Допускает любую 3.x версию
sqlalchemy>=2.0.0 # Может сломаться при обновлении
```
**Рекомендация:** Зафиксировать точные версии (`==3.4.1`), использовать Poetry или Pipenv.

#### P2: Rate limiter fallback не thread-safe
```python
_storage = {}  # In-memory dict, не защищён от race conditions в async
```

#### P2: N+1 запросы
При загрузке списка смен с исполнителями нет `joinedload()` / `selectinload()`. Каждый исполнитель загружается отдельным запросом.

#### P2: Тестовое покрытие
- 58 файлов тестов, но нет coverage-отчёта
- Отсутствуют тесты на: обработчики, миграции, ML-скоринг
- Нет pytest fixtures для типовых сценариев

### 3.5 Низкий приоритет

- Нет стандартизации формата API-ответов FastAPI
- Отсутствуют security headers (`X-Content-Type-Options`, `X-XSS-Protection`)
- Debug=True по умолчанию в settings.py

## 4. План интеграции InfraSafe + UK Management Bot

### 4.1 Архитектура интеграции

```
                    +------------------+
                    |   PostgreSQL     |
                    | (shared or sync) |
                    +--------+---------+
                             |
              +--------------+--------------+
              |                             |
     +--------v--------+          +--------v--------+
     |   InfraSafe      |          |   UK Bot         |
     |   (Node.js)      |          |   (Python)       |
     |                  |          |                  |
     | Sensors/Metrics  | -------> | Request Mgmt    |
     | Maps/Dashboard   |  alerts  | Shift Mgmt      |
     | Alerts Engine    |          | ML Assignment    |
     +--------+---------+         +--------+---------+
              |                             |
              v                             v
     +--------+---------+          +--------+---------+
     |  Web Dashboard   |          |  Telegram Bot    |
     |  (Leaflet Maps)  |          |  (Aiogram)       |
     +-----------------+          +-----------------+
```

### 4.2 Сценарии интеграции

#### Сценарий A: Alert -> Request (автоматическое создание заявок)

**Поток данных:**
1. InfraSafe обнаруживает аномалию (напряжение < 200В, протечка, давление < 1 бар)
2. Создаёт alert в таблице `alerts`
3. **Интеграционный сервис** (новый) создаёт заявку в UK Bot:
   - Категория маппится: electricity -> "Электрика", water -> "Сантехника", heat -> "Отопление"
   - Адрес маппится: `buildings.address` -> `apartments.building_id`
   - Срочность определяется по severity алерта
4. UK Bot назначает исполнителя через SmartDispatcher
5. Статус заявки обновляется в обоих системах

**Необходимые изменения:**
- InfraSafe: Webhook/API endpoint для отправки алертов
- UK Bot: API endpoint для создания заявок из внешних систем
- Shared: Маппинг зданий между системами (общий `building_id` или таблица соответствий)

#### Сценарий B: Shared Building Directory (общий справочник зданий)

**Проблема:** InfraSafe хранит здания с координатами (PostGIS), UK Bot хранит адресную иерархию (Дворы -> Здания -> Квартиры).

**Решение:** Расширить таблицу `buildings` в UK Bot координатами из InfraSafe, или создать mapping-таблицу:
```sql
CREATE TABLE building_mapping (
    infrasafe_building_id INTEGER REFERENCES infrasafe.buildings(building_id),
    uk_building_id INTEGER REFERENCES uk.buildings(id),
    PRIMARY KEY (infrasafe_building_id, uk_building_id)
);
```

#### Сценарий C: Dashboard Integration (UK данные на карте InfraSafe)

**Поток данных:**
1. InfraSafe Dashboard показывает здания на карте
2. При клике на здание, popup показывает:
   - Метрики датчиков (из InfraSafe)
   - Активные заявки (из UK Bot)
   - Текущий исполнитель на смене (из UK Bot)
3. Карта отображает дополнительный слой "Заявки" с маркерами

**Необходимые изменения:**
- UK Bot: REST API endpoint `/api/requests?building_id=X&status=active`
- UK Bot: REST API endpoint `/api/shifts/current?building_id=X`
- InfraSafe: Новый overlay-слой "Заявки УК" в map-layers-control
- InfraSafe: Расширение popup зданий блоком "Активные заявки"

#### Сценарий D: Unified Analytics (объединённая аналитика)

Создание единого аналитического дашборда:
- Корреляция: частота алертов vs количество заявок по зданию
- Эффективность: среднее время от алерта до закрытия заявки
- Предиктив: ML-модель прогнозирует поломки на основе метрик + истории заявок

### 4.3 Технические варианты интеграции

#### Вариант 1: API Gateway (рекомендуется)
```
Client -> Nginx -> /api/infrasafe/* -> Express.js (InfraSafe)
                -> /api/uk/*        -> FastAPI (UK Bot)
                -> /api/integrated/* -> Integration Service
```
**Плюсы:** Чистое разделение, независимое развёртывание, единый CORS.
**Минусы:** Дополнительный сервис для маршрутизации.

#### Вариант 2: Event-Driven (через PostgreSQL LISTEN/NOTIFY)
```sql
-- InfraSafe создаёт алерт
INSERT INTO alerts (...) VALUES (...);
-- PostgreSQL trigger отправляет уведомление
NOTIFY new_alert, '{"alert_id": 123, "building_id": 5, "severity": "critical"}';
-- UK Bot слушает канал и создаёт заявку
LISTEN new_alert;
```
**Плюсы:** Минимум изменений, реальное время, нет дополнительных сервисов.
**Минусы:** Привязка к одной БД, сложнее дебажить.

#### Вариант 3: Message Queue (Redis/RabbitMQ)
```
InfraSafe -> Redis PubSub -> UK Bot Integration Worker
```
**Плюсы:** Надёжная доставка, буферизация, масштабирование.
**Минусы:** Дополнительная инфраструктура (Redis уже есть в UK Bot).

## 5. Риски и ограничения

### 5.1 Технические риски

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Разные схемы адресов в системах | Высокая | Высокое | Маппинг-таблица + ручная верификация |
| Telegram Bot rate limits (30 msg/sec) | Средняя | Среднее | Очередь уведомлений, batch-отправка |
| Разные стеки усложняют поддержку | Средняя | Среднее | Документация API-контрактов, интеграционные тесты |
| PostgreSQL version mismatch | Низкая | Низкое | Обе системы на PostgreSQL 15+ |

### 5.2 Организационные риски

- **Разный стек (JS + Python):** Нужны разработчики обоих стеков или full-stack
- **Два deployment pipeline:** Docker Compose для каждого, нужна координация
- **Разная модель данных пользователей:** InfraSafe (JWT) vs UK Bot (Telegram ID)

### 5.3 Ограничения

- UK Bot привязан к Telegram -- невозможно масштабировать горизонтально (один bot token = один процесс)
- InfraSafe не имеет WebSocket -- обновления только через polling
- Нет общей системы аутентификации -- потребуется SSO или federation

## 7. Заключение

InfraSafe и UK Management Bot -- **комплементарные системы**, покрывающие разные этапы управления инфраструктурой:

- **InfraSafe** = *автоматическое обнаружение проблем* (датчики, метрики, алерты)
- **UK Bot** = *человеческое решение проблем* (заявки, исполнители, приёмка)

Интеграция создаст **замкнутый цикл**: обнаружение -> заявка -> назначение -> исполнение -> подтверждение -> мониторинг. Ключевая точка соединения -- **Alert -> Request pipeline**, который автоматизирует создание заявок по данным датчиков.

Оба проекта используют PostgreSQL, что упрощает интеграцию. Основные вызовы: различие адресных моделей, разные стеки (JS/Python), и необходимость унификации аутентификации.

Рекомендуемый первый шаг: исправить P0 уязвимости в UK Bot, затем реализовать минимальный API для обмена данными о зданиях и заявках.
