# InfraSafe — Архитектурный дебат: оптимизация сервисного стека

## Контекст проекта

InfraSafe — IoT/smart-building платформа для управления инженерными системами зданий. Платформа работает на edge/on-prem железе, поддерживается командой 1–2 человека. Целевой масштаб: от 10 до 500 IoT-устройств (контроллеры, датчики, актуаторы).

## Текущий data pipeline

```
Контроллеры (физическая установка, автономная работа)
       │
       │ контроллер сам инициирует WireGuard-туннель до ядра
       │ (reverse connection — мы не подключаемся к ним, они поднимают сессию к нам)
       ▼
┌─────────────────┐
│   WireGuard     │ ← входная точка, все контроллеры стучатся сюда
│   VPN Gateway   │
└────────┬────────┘
         │ MQTT
         ▼
┌─────────────────┐         ┌─────────────────┐
│   Mosquitto     │────────▶│   InfluxDB      │  time-series хранение
│   MQTT Broker   │         │                 │  (сырые показания датчиков)
└────────┬────────┘         └─────────────────┘
         │ MQTT subscribe
         ▼
┌─────────────────┐
│   Node-RED      │  бизнес-логика, правила, обработка событий
│   Rule Engine   │
└────────┬────────┘
         │ обработанные результаты
         ▼
┌─────────────────┐         ┌─────────────────┐
│   PostgreSQL    │────────▶│   Frontend      │  визуализация для пользователей
│   (основная БД) │         │   (Web UI)      │
└─────────────────┘         └─────────────────┘

┌─────────────────┐
│   Grafana       │  ← присутствует в стеке, роль размыта
│   (дашборды?)   │
└─────────────────┘
```

### Важное уточнение по WireGuard

Контроллеры устанавливаются физически на объекте. После установки контроллер **сам поднимает WireGuard-сессию до ядра платформы** (reverse connection pattern). Мы не инициируем подключение к контроллерам — они за NAT, файрволами, на мобильных сетях. WireGuard используется именно как обратный канал: контроллер стучится к нашему серверу и поднимает persistent tunnel, через который далее идёт MQTT-трафик и, возможно, управляющие команды.

## Раунд 1 — Защита текущей архитектуры

**Агент: `@industrial-automation-engineer`**

Проанализируй текущий IoT data pipeline InfraSafe с позиции **защитника текущей архитектуры**. Твоя задача — обосновать выбор каждого компонента.

### Что обосновать

**Mosquitto (MQTT Broker)**
- Почему Mosquitto, а не EMQX, NanoMQ, HiveMQ?
- Footprint на edge-железе, стабильность, зрелость
- Достаточен ли для 500 устройств с QoS 1/2?
- MQTT-bridge для мульти-сайтовых деплоев — нужен ли?

**InfluxDB (Time-Series Storage)**
- Зачем отдельная time-series БД если финал в PostgreSQL?
- Retention policies, downsampling, continuous queries — используем ли мы это?
- Скорость записи потоковых данных: InfluxDB vs PostgreSQL напрямую
- Какой объём данных генерируют контроллеры (samples/sec)?

**Node-RED (Rule Engine)**
- Преимущества visual flow engine для IoT-правил и автоматизации
- Скорость прототипирования новых сценариев
- Интеграция с MQTT (нативные ноды)
- Community ноды для промышленных протоколов (Modbus, BACnet, OPC UA)

**Grafana**
- Есть ли у Grafana уникальная роль, которую не закрывает фронтенд?
- Ops-мониторинг vs клиентская визуализация — разные задачи?
- Alerting через Grafana — используется ли?

**WireGuard (Reverse VPN)**
- Reverse connection pattern — контроллер инициирует туннель
- Минимальный overhead, работа за NAT, на мобильных сетях (LTE/5G)
- Автоматическое provisioning ключей при масштабировании до 100+ устройств
- Persistent keepalive для поддержания туннеля через NAT
- Альтернативы: Tailscale (WireGuard-based), Nebula, ZeroTier — оправдана ли замена?

### Формат ответа
Для каждого сервиса:
- Что конкретно потеряем если уберём
- Какие сценарии он закрывает лучше альтернатив
- Какие ограничения у текущего выбора

## Раунд 2 — Критика и реорганизация

**Агент: `@senior-architect-refactor`**

Проведи критический анализ того же pipeline с позиции **архитектора-реформатора**. Твоя задача — найти лишние звенья, упростить стек, сократить операционную нагрузку.

### Направления атаки

**InfluxDB + PostgreSQL → TimescaleDB?**
- TimescaleDB — расширение PostgreSQL, закрывает оба кейса в одном сервисе
- Continuous aggregates заменяют continuous queries InfluxDB
- Compression, retention policies — всё есть
- Минус один сервис = минус один backup, минус одна точка отказа, минус RAM
- Что конкретно теряем при отказе от InfluxDB? Flux-язык запросов? Нужен ли он нам?

**Node-RED — актив или техдолг?**
- Flow-файлы: JSON, но git-diff по ним нечитаем
- Как версионировать flows? Как делать code review?
- Отладка в продакшене — console.log в лучшем случае
- При усложнении бизнес-логики flows превращаются в спагетти
- Альтернатива 1: TypeScript-сервис с правилами в коде (тестируемо, версионируемо)
- Альтернатива 2: Benthos / Vector для stream processing
- Альтернатива 3: оставить Node-RED, но формализовать — flows as code, CI/CD
- При какой сложности логики Node-RED становится проблемой?

**Grafana — убрать или переосмыслить?**
- Фронтенд уже визуализирует из PostgreSQL — Grafana дублирует?
- Если нужен только alerting — вынести в отдельный сервис (alertmanager, или логика в Node-RED/TypeScript)
- Если нужны ops-дашборды — оставить, но чётко разграничить: Grafana = ops, Frontend = клиенты
- RAM/CPU footprint Grafana на edge-железе

**Mosquitto → EMQX?**
- EMQX rule engine может маршрутизировать данные напрямую в PostgreSQL/TimescaleDB
- Потенциально заменяет часть Node-RED (простая маршрутизация, фильтрация)
- Webhook интеграции из коробки
- НО: footprint EMQX значительно больше Mosquitto
- НО: оправдана ли сложность для команды 1–2 человека?
- НО: нужен ли нам кластер MQTT? Или один Mosquitto справляется?

**WireGuard — архитектурные вопросы масштабирования**
- 100+ контроллеров = 100+ WireGuard peers на одном интерфейсе
- Автоматическое provisioning: как генерировать и доставлять ключи на контроллер при физической установке?
- Мониторинг состояния туннелей (какой контроллер оффлайн?)
- Tailscale/Headscale поверх WireGuard — упрощает ли управление ключами?

**Главный вопрос**
6 сервисов (WireGuard, Mosquitto, InfluxDB, Node-RED, PostgreSQL, Grafana) плюс фронтенд — это слишком много для 1–2 человек на edge-железе? Какой минимальный стек закроет 90% функциональности?

### Формат ответа
- Конкретные предложения по замене/объединению сервисов
- Для каждого предложения: что выигрываем, что теряем, стоимость миграции
- Целевая архитектура с минимальным числом сервисов
